前言

简单总结下API开放平台里面每个功能模块的流程，以及涉及到的技术，为后期拓展功能做准备。

技术栈

前端：React + Ant Design Pro + OpenAPI前端代码生成

后端：Spring Boot、 MySQL、 MyBatis Plus 、 Spring Boot Starter、 Spring Cloud Gateway、 Dubbo分布式

功能实现

登录注册

对于登录注册，前端传输的是明文信息，即密码和账号都是没有加密的

那么这样就会有安全性问题，比如说前端传输的报文被抓包了，那么用户的账号信息就存在很大风险

因此，后端 注册 采用了 盐值 + 原密码 利用md5算法加密的方法，作为存放在数据库中的密码。

盐值方便混淆加密前的输入。

但又有一个新的问题出现，登录的时候，怎么确认密码是对的呢？

其实办法也很简单，就是重现一遍登录注册的操作（盐值 + 明文密码 再md5加密一遍），然后去和数据库中存放的加密数据进行比对，如果一致，那么登录成功。

值得注意的是，用户的登录状态是保存在session中，那么为什么不保存在cooki中呢？

主要考虑了数据存放位置、安全程度、以及数据存储大小的问题

  1. 数据存放位置。cookie数据存放在客户的浏览器上，session数据放在服务器上。
  2. 安全程度不同。cookie不是很安全，别人可以分析存放在本地的COOKIE并进行COOKIE欺骗,考虑到安全应当使用session。
  3. 单个cookie保存的数据不能超过4K，很多浏览器都限制一个站点最多保存20个cookie，而session则存储与服务端，浏览器对其没有限制。

这里有一个改进的点，如果追求性能的话，可以先从缓存中找用户id，再找数据库

这里补充一下，VO对象和DTO对象的区别，也就便于理解为什么要封装返回VO对象，而不是直接返回查询到的数据？

  • 设计目的：VO主要用于封装数据，保证数据的完整性；DTO用于在不同层之间传输数据，提高系统性能。
  • 可变性：VO通常是不可变的，一旦创建就不能修改；DTO是可变的，可以根据业务需求灵活调整。
  • 属性和方法：VO只包含属性的get方法，没有set方法；DTO包含setter和getter方法，方便数据传输和修改。
  • 适用场景：VO适用于实体对象的封装和展示层的数据展示；DTO适用于服务层和展示层之间的数据传输。

用户鉴权

由于系统中，有些功能如接口调用分析，只能允许给管理员调用，而不允许展示给用户看，因此需要在方法级别上进行权限校验。

采用的方法：自定义注解

解析一下这个注解的主要元素：

  1. @Target(ElementType.METHOD): 这个注解告诉编译器，AuthCheck 注解只能被应用于方法（Method）上。
  2. @Retention(RetentionPolicy.RUNTIME): 这个注解告诉编译器，这个注解将在运行时保留，这是因为你可能需要在运行时通过反射来获取注解信息。
  3. String[] anyRole() default "";: 这是一个注解的属性，名为 anyRole，是一个字符串数组。这个属性用于指定在进行权限校验时，需要具备的任何一个角色。默认值是空数组，表示不需要任何特定角色。
  4. String mustRole() default "";: 这是另一个注解的属性，名为 mustRole，是一个字符串。这个属性用于指定在进行权限校验时，必须具备的特定角色。默认值是空字符串，表示不需要特定角色。

通过添加@AuthCheck 注解，并在注解的属性中指定相应的角色，可以灵活地定义在调用这个方法时需要满足的权限条件。

API签名认证

为什么需要设计API签名认证？

  • 保证安全性，不能让一个接口随意一个用户都能调用
  • 适用于不保存登录状态的场景。只认签名，不关注用户态

实现方式：

通过http request header 传递参数

  • 参数1 accessKey ： 调用的标识 （类似于用户名）
  • 参数2 secretKey : 密钥 这个参数不能放在请求头中 (类似于密码)
  • 参数3 用户请求参数
  • 参数4 签名
  • 参数5 nonce 随机数， 确保当前的签名只能用一次
  • 参数6 timestamp 时间戳， 校验时间戳是否过期

区别是，ak和sk是无状态的

引入参数5和参数6的作用是什么？

随机数和时间戳是为了防重放，增加安全性。

补充：重放攻击是一种网络攻击，攻击者通过拦截和重新发送合法的通信会话，试图欺骗系统。在API中，如果不采取措施来防止重放攻击，攻击者可能会通过多次发送相同的请求来欺骗系统，执行一些不必要的或者恶意的操作。

签名的生成方式是什么？

将请求头中获取到的密钥 利用SHA256进行非对称机密，去生成签名。采用非对称是因为这种方式是不可解密的，更为安全

怎么判断签名对不对？

服务器用一模一样的参数和算法去生成签名，只要和用户传递的一致，就表示一致

提供SDK

作为一个API开放平台，为开发者提供一个SDK进行二次开发也是有必要的，这样能促进平台生态系统的发展，同时，在二次开发过程中，SDK让开发者无需从头开始编写复杂的代码，能够更快地开发应用程序。

为了让开发者能更方便的使用，采用starter的实现方式。

starter的好处在于，不需要开发者手动创建客户端，只需要引入依赖，编写配置文件，就能有starter封装好的接口和方法

API网关

API网关的作用是什么？

试想一下，假如我们要对每个调用的接口进行次数统计，我们可以在每个接口的代码处增加调用逻辑的代码，但是这样做，每个项目都要增加逻辑，以后维护还每一个地方都要修改，那么为什么不学车站那样，统一弄个验票口呢？

因此，网关的需求就出现了，我们可以设计一个进行统一处理的网关，让用户把接口接入到网关中，并让网关能够找到并调用，在网关处进行次数统计即可

网关其实还能做很多功能，更多的话，可以去看spring cloud gateway的文档

这里介绍一下项目里面使用到的功能：

  • 路由。简单来说就是请求转发的意思。
  • 统一鉴权。比如判断用户是否有权限进行操作。对于接口的访问，可以在网关处进行统一判断，而不需要在每个接口项目里面进行鉴权
  • 统一日志。
  • 统一的业务处理。比如接口的调用次数统计，将这类通用逻辑放到网关层面。

实现主要采用的是spring cloude gateway来实现

RPC调用

我们希望在网关中能直接调用其他项目的接口，这里面可以需要用到RPC（远程过程调用）

当然，调用其他项目的方法有很多:

  • 复制代码和依赖，环境
  • HTTP请求，提供一个接口，给其他项目调用
  • RPC
  • 把公共的代码打成jar包，其他项目去引用

这里，我们采用了RPC的方法。

RPC：像调用本地方法一样去调用远程方法

和直接调用HTTP的区别：

  1. 对开发者更透明，不需要像HTTP请求那样进行额外的封装，按照客户端工具类的那种方式去发送请求，而是像调用本地方法那样。
  2. RPC向远程服务器发送请求时，可以选择很多其他的协议，而不只局限于HTTP协议，这样可能性能更高

RPC调用模型

概念补充

  • 提供者：为其他人提供已经写好的代码，让其他人可以使用
  • 调用方：一旦服务提供者提供了服务，调用方需要能够找到这个服务的位置。
  • 存储（注册中心）：用于存储提供者提供的方法信息。调用方可以从存储中获取提供者的地址和方法信息。

在项目里面，RPC的实现，我们采用了Dubbo框架，而注册中心我们采用的是nacos
