JWT学习笔记

发表于2024-03-12|更新于2024-03-14
|字数总计:1.2k|阅读时长:3分钟|阅读量:

JWT的概念及其特点

JWT （JSON Web Token）是一种基于 Token 的认证授权机制。 从 JWT 的全称可以看出，JWT 本身也是 Token，一种规范化之后的 JSON 结构的 Token。

设计目的：不需要服务器端存储状态，安全地传递非敏感信息

JWT 本质上就是一组字串，通过（.）切分成三个为 Base64 编码的部分：

  • Header : 描述 JWT 的元数据，定义了生成签名的算法以及 Token 的类型。
  • Payload : 用来存放实际需要传递的数据
  • Signature（签名）：服务器通过 Payload、Header 和一个密钥(Secret)使用 Header 里面指定的签名算法（默认是 HMAC SHA256）生成。

JWT 通常是这样的：xxxxx.yyyyy.zzzzz。

Payload 部分默认是不加密的，一定不要将隐私信息存放在 Payload 当中！！！

在 JWT 的 JWS 实现方式中，它的头部和载荷是没有加密的，所以它的目的不是为了隐藏数据，是想防止数据被篡改，这点是通过第三部分签名来实现的。

JWT的应用场景

1、用户身份认证。（最普遍的用途）

这里的 JWT 加密 一般是用对称加密的方式

一般步骤如下：

  • 用户向服务器发送用户名、密码以及验证码用于登陆系统。
  • 如果用户用户名、密码以及验证码校验正确的话，服务端会返回已经签名的 Token，也就是 JWT。
  • 用户以后每次向后端发请求都在 Header 中带上这个 JWT 。
  • 服务端检查 JWT 并从中获取用户相关信息。

2、密码重置和电子邮件认证

一般步骤如下：

用户发起密码重置或者电子邮件认证时，服务器会生成包含用户信息的JWT并组成一个链接发往用户的邮箱，用户点击后就能进行密码重置或者电子邮件认证

JWT与其他认证方式的对比

1.cookie来传递用户标识

2.基于API Key的认证方式。（基于非对称加密）

该方式，API开放平台，云服务商提供的接口服务 用的多

JWT 存在的优点和缺点

优点：

  1. 无状态。服务器不需要存储 Session 信息。这显然增加了系统的可用性和伸缩性，大大减轻了服务端的压力。
  2. 避免了CSRF（跨域请求伪造）。在我们登录成功获得 JWT 之后，一般会选择存放在 localStorage 中。整个过程压根不会涉及到 Cookie。因此可以避免 CSRF 攻击。

一些争议：JWT有无状态是如何定义的？

主流说法是，JWT是无状态的，因为服务器只校验jwt的合法性，完全不去查Redis，数据库，查了才是有状态

还有一种声音是，，jwt 是有状态的，是说 JWT 本身就携带了状态信息，不需要额外去查 redis 或者数据库。比如我们在说 HTTP 是无状态的协议，是因为 HTTP 的协议本身并不维护会话信息

缺点：

  1. 服务端没法真正退出登录。因为那个签发的 token 还在有效期内。
  2. 用户信息修改没法及时同步。比如用户的 name 修改了，或者用户的角色变化了，从原来的 admin 改成了普通用户，但是之前签发的 JWT token 还是生效状态，里面的信息还是修改前的
  3. Token泄露无有效手段立马将token无效。

JWT是如何进行身份验证的？

步骤如下：

  1. 用户向服务器发送用户名、密码以及验证码用于登陆系统。
  2. 如果用户用户名、密码以及验证码校验正确的话，服务端会返回已经签名的 Token，也就是 JWT。
  3. 用户以后每次向后端发请求都在 Header 中带上这个 JWT 。
  4. 服务端检查 JWT 并从中获取用户相关信息。

两点建议：

  1. 建议将 JWT 存放在 localStorage 中，放在 Cookie 中会有 CSRF 风险。
  2. 请求服务端并携带 JWT 的常见做法是将其放在 HTTP Header 的 Authorization 字段中

JWT是如何防止被篡改的

有了签名之后，即使 JWT 被泄露或者截获，黑客也没办法同时篡改 Signature、Header、Payload。

这是为什么呢？因为服务端拿到 JWT 之后，会解析出其中包含的 Header、Payload 以及 Signature 。服务端会根据 Header、Payload、密钥再次生成一个 Signature。拿新生成的 Signature 和 JWT 中的 Signature 作对比，如果一样就说明 Header 和 Payload 没有被修改。

JWT 的核心在于签名，签名的核心在于服务器的密钥不能泄露
